## [CRITICAL] 1. POST /api/route-stops performs an unscoped cross-tenant delete by jobId
- **Categoria:** tenant-isolation | **Confianza:** high
- **Archivo:** `src/app/api/route-stops/route.ts`
- **Detalle:** CONFIRMED. POST /api/route-stops (src/app/api/route-stops/route.ts) authenticates, asserts route_stop:create, and derives tenantCtx correctly, but line 58 runs an unscoped destructive delete: `db.delete(routeStops).where(eq(routeStops.jobId, jobId))` with no companyId predicate. jobId is taken verbatim from the request body (line 25) and never verified to belong to tenantCtx.companyId. setTenantContext is plain AsyncLocalStorage (no query interception) and withTenantFilter is opt-in and not applied here (it IS applied in the GET handler), so there is no fallback scoping; routeStops.companyId exists, so scoping is possible. Any holder of route_stop:create who knows another tenant's job UUID can wipe that tenant's route stops (cross-tenant destructive IDOR). Practical exploitability is gated by the jobId being a non-enumerable UUID, but the authorization gap is real. Separately, the insert stamps companyId from tenantCtx but copies routeId/orderId/vehicleId/userId from the body with only presence validation (lines 34-55, 67-70), so inserted stops can reference another company's order/vehicle/driver. Fix: scope the delete with `and(eq(routeStops.jobId, jobId), eq(routeStops.companyId, tenantCtx.companyId))` (or via withTenantFilter), and validate that jobId and each stop's routeId/orderId/vehicleId/userId belong to tenantCtx.companyId before inserting, returning 403/404 otherwise.
- **Evidencia:** Line 58: await db.delete(routeStops).where(eq(routeStops.jobId, jobId)); — jobId from `const { jobId, stops } = body;` (line 25), no companyId filter.
- **Fix:** Scope the delete with the tenant filter: .where(and(eq(routeStops.jobId, jobId), eq(routeStops.companyId, tenantCtx.companyId))). Before inserting, verify jobId (and each stop's routeId/orderId/vehicleId/userId) belongs to tenantCtx.companyId, returning 404/403 otherwise.

## [HIGH] 2. GET /api/metrics/history derives companyId from an unset AsyncLocalStorage store instead of validating it
- **Categoria:** tenant-isolation | **Confianza:** high
- **Archivo:** `src/app/api/metrics/history/route.ts`
- **Detalle:** GET /api/metrics/history (src/app/api/metrics/history/route.ts) enforces metrics:read via requireRoutePermission but then reads companyId from the ambient AsyncLocalStorage store (getTenantContext() from @/db/tenant-aware, line 25) without re-deriving/validating tenant in the handler. requireRoutePermission does not populate that store. The store IS set per-request by src/proxy.ts (the Next 16 middleware) using a JWT-authoritative effectiveCompanyId, but it does so via AsyncLocalStorage.enterWith() in the middleware context, which the project's own code documents as unreliable for propagation into App Router handlers (tenant-aware.ts:18-23). Every robust sibling handler (e.g. orders/route.ts, output/route.ts) therefore re-calls setTenantContext inside the handler; this route does not. Consequence: getTenantContext().companyId is fragile — typically null (→400 "Company context required"), but if a prior request on the same worker left a value in ALS via its own in-handler setTenantContext, this route can serve another tenant's historical metrics (getHistoricalMetrics/getMetricsSummaryStats). Note the finding's claims that this is a unique deviation and that no JWT/header validation exists are inaccurate — a co-defective sibling (optimization/jobs/[id]/metrics/route.ts) shares the pattern, and the proxy does perform JWT-authoritative tenant resolution. Fix as recommended: in the handler, `const tenantCtx = extractTenantContextAuthed(request, authResult); if (tenantCtx instanceof NextResponse) return tenantCtx; setTenantContext(tenantCtx);` and use tenantCtx.companyId — and apply the same fix to optimization/jobs/[id]/metrics/route.ts.
- **Evidencia:** Line 25: const tenantContext = getTenantContext(); then line 45/66 pass tenantContext.companyId to getHistoricalMetrics/getMetricsSummaryStats. No extractTenantContextAuthed/setTenantContext anywhere in the file.
- **Fix:** Replace getTenantContext() with `const tenantCtx = extractTenantContextAuthed(request, authResult); if (tenantCtx instanceof NextResponse) return tenantCtx; setTenantContext(tenantCtx);` and use tenantCtx.companyId — matching the documented pattern used by sibling routes.
